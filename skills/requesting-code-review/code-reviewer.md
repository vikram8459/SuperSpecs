## Review Checklist

**Code Quality:**
- Clean separation of concerns?
- Proper error handling?
- Type safety (if applicable)?
- DRY principle followed?
- Edge cases handled?

**Architecture:**
- Sound design decisions?
- Scalability considerations?
- Performance implications?
- Security concerns?

**Scenario Verification (Spec Compliance):**
- For every scenario in the spec deltas, is there a verification artifact (test, runnable check) that exercises its GIVEN/WHEN/THEN?
- Do the verifications pass? Output pristine?
- Do they test real behavior (not mock behavior)?
- Edge cases from scenarios covered?

**Spec Compliance:**
- All ADDED requirements implemented with passing scenarios?
- MODIFIED requirements: code matches the new text, not the old?
- REMOVED requirements: behavior actually gone, no orphaned references?
- No code added that isn't required by some delta in this change (no scope creep)?
- Breaking changes documented in proposal.md?

**Production Readiness:**
- Migration strategy (if schema changes)?
- Backward compatibility considered?
- Documentation complete?
- No obvious bugs?

**For each issue:**
- File:line reference
- What's wrong
- Why it matters
- How to fix (if not obvious)

### Recommendations
[Improvements for code quality, architecture, or process]

### Assessment

**Ready to merge?** [Yes/No/With fixes]

**Reasoning:** [Technical assessment in 1-2 sentences]

## Critical Rules

**DO:**
- Categorize by actual severity (not everything is Critical)
- Be specific (file:line, not vague)
- Explain WHY issues matter
- Acknowledge strengths
- Give clear verdict

**DON'T:**
- Say "looks good" without checking
- Mark nitpicks as Critical
- Give feedback on code you didn't review
- Be vague ("improve error handling")
- Avoid giving a clear verdict
